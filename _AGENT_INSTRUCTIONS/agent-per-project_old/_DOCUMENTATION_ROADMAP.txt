Here’s a stepwise plan to turn _AGENT_INSTRUCTIONS into a reliable, self-healing documentation system that stays useful as the project evolves.

OVERARCHING PRINCIPLE: move all useful documentation currently sitting within _AGENT_CONTEXT to _AGENT_INSTRUCTIONS (agent-instructions-shared and agent-per-project).
When this is complete, retire _AGENT_CONTEXT and wire .github/copilot-instructions.md directly to _AGENT_INSTRUCTIONS.

1. Establish the Backbone

Inventory & Audit: list every existing note in _AGENT_INSTRUCTIONS and legacy _AGENT_CONTEXT, mark each as current/obsolete/needs merge.
Canonical TOC: land a living “map” (e.g., _AGENT_INSTRUCTIONS/INDEX.md) that mirrors the folder tree and links to every canonical doc. This becomes the mandatory starting point for new agents.
Shelve Legacy Notes: move superseded _AGENT_CONTEXT material into an archive or import the relevant pieces into the new structure so there’s only one source for each fact.
2. Standardize Formats

Adopt short frontmatter on every file (tier, scope, owner, last_reviewed, source_of_truth, links). Require it for new docs and retrofit old ones gradually.
Define doc types (overview, playbook, reference, ADR, SOP) and keep reusable templates in agent-instructions-shared/90-templates. Add a README that shows when to use each template.
Require “Explicit scope + last_reviewed” so automation can flag stale notes.
3. Create the Core Knowledge Pods

For each major concern (environment setup, data flow, import pipeline, schema, navigation, UI patterns, testing):
Nominate a single canonical doc within the right subfolder (e.g., 00-project, 10-features, 20-migrations, 40-integration).
Summarize the topic in that doc, link out to supporting notes, and mark legacy alternatives as deprecated.
Example pods:
Environment & tooling: 00-project/env-and-secrets.md, 40-integration/services-and-keys.md.
Databases: data-locations.md, 20-migrations/schema-reference.md, migration playbooks.
Import pipeline (Rust extractor, chat.db, AddressBook): probably a new 40-integration/import-pipeline.md that references schema docs and Rust service in 10-features/....
UI/navigation: 10-features/chats/messages/... reorganized around ViewSpec, timeline behavior, etc.
ADR index: ensure 30-decisions/ has an index linking each ADR to the subsystem it governs.
4. Define Maintenance Process

Document how docs get updated: e.g., new feature → add/update doc + link; schema change → update schema-reference.md + bump last_reviewed.
Add a simple checklist for PRs that touch core systems (code + docs updated, index entry touched).
Optionally add a script or GitHub Action to warn when last_reviewed is older than X months.
5. Content Build-Out Roadmap

Maintain a living backlog (“Documentation TODOs”) that lists missing or stale topics (Rust extractor, timeline logic, Riverpod patterns, etc.).
For each sprint/feature, pick a few items:
Capture the canonical doc (using templates).
Link it from the TOC and relevant pods.
Remove or update any conflicting notes.
6. Cross-Referencing & Discoverability

Encourage forward/back links: each doc’s “Related” section points to other pods.
Update README files at each folder level to explain what lives beneath and how to use it.
Ensure the canonical TOC references these folder READMEs.
7. Automation (optional but valuable)

Linters or scripts to ensure new docs have frontmatter, TOC inclusion, valid links.
Scheduled review reminders for docs whose last_reviewed is stale.
Executing this plan step-by-step lets us bring in topics like the Rust attributedBody extractor without creating isolated notes—each addition fits into the larger map, gets linked, and inherits dating/ownership so it stays current.
