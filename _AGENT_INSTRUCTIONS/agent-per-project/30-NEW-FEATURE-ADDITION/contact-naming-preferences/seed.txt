This is a really good problem to be thinking about now, because you’re right: naming is identity, and identity tweaks should feel local, obvious, and reversible, not like “configuration work”.

I’ll structure this in four layers:
	1.	Core mental model (what the user thinks they’re doing)
	2.	Global vs per-contact settings (how power scales without confusion)
	3.	Where editing should live (Chooser vs Hero vs Settings)
	4.	Concrete UX recommendations (including what not to do)

I’ll refer explicitly to A / B / C as you framed them.

⸻

1. The right mental model (this is the key)

The user is never thinking:

“I want to configure contact short-name policies.”

They are thinking:

“I want this person to be called this.”

Everything else should support that.

That implies a hierarchy:
	•	Primary action: Rename this contact (nickname / short name)
	•	Secondary action: Decide how names are generally formed
	•	Tertiary action: Bulk review or cleanup

Once you align with that, the design mostly falls out naturally.

⸻

2. Global vs per-contact: keep it asymmetric

Your current Settings panel (A) is already on the right track, but I’d make the asymmetry more explicit.

Recommended model

Global setting (default behavior)
Located in Settings → Contact Short Names:
	•	⦿ First name only
	•	⦿ First initial + last name
	•	⦿ Nickname when available (fallback to first name)

This answers:

“If I haven’t thought about this person yet, what should I see?”

That’s good.

Per-contact override (exception handling)
Each contact has:
	•	Display name mode:
	•	Inherit global default
	•	Force first name
	•	Force initial + last
	•	Force nickname
	•	Nickname (only relevant if nickname is chosen or preferred)

This answers:

“This person is special.”

Crucially:
Most contacts will inherit. Only a few will be overridden.

This keeps your data model clean and your UI sparse.

⸻

3. Where editing should happen (A vs B vs C)

A. Global Settings panel (good, but not sufficient)

Your Settings screen (A) is doing the right job but should not be the primary place users rename people.

It’s fine for:
	•	Choosing the default rule
	•	Reviewing existing nicknames (advanced users)
	•	Bulk cleanup (“why does ICE look weird?”)

But as you already sense:

this is not how humans think about names.

So: keep it, but de-emphasize it.

⸻

B. Chooser panel: resist inline editing here

Your instinct is correct to be cautious.

The chooser is a navigation surface, not a configuration surface.

Problems with inline editing in the chooser:
	•	Hover affordances are invisible until discovered
	•	Text fields in lists create visual noise
	•	It encourages accidental edits
	•	It scales badly with long lists

Recommendation:
❌ Do not allow direct editing in the chooser list itself.

Instead:
	•	Allow a subtle affordance like:
	•	Context menu: “Edit contact display name…”
	•	Or a small pencil icon only on the selected row

But don’t put text fields there.

⸻

C. Hero card: this is the right place 👍

Your Hero panel (C) is the perfect location for this.

Why it works:
	•	The user has already said “I care about this person”
	•	There’s visual breathing room
	•	Changes feel contextual, not global
	•	It avoids cluttering navigation

This matches your stated goal beautifully:

apparent, intuitive, and non-disruptive

⸻

4.3 Settings panel: advanced & global

Your existing Settings UI is mostly right, but I’d tweak framing:

Rename section title
From:

Contact short names

To something more human:

How names are shown

Clarify scope
Add helper text like:

This sets the default for new and unchanged contacts.
Individual contacts can override this.

That single sentence prevents confusion later.

⸻

5. One subtle but important suggestion

Don’t call it “short name” in the UI

It’s fine internally, but externally:
	•	“Short name” sounds technical
	•	Users think in terms of “what I call them”

Better labels:
	•	Display name
	•	Shown name
	•	Name used in lists

Internally you can still call it shortName.

⸻

6. Why this works with your app specifically

Given Remember This Text’s emotional and archival nature:
	•	Names are part of memory
	•	Renaming someone is intimate
	•	Users will do it in the moment, not in batch mode

Your Hero-centric approach respects that psychology.

You’re not building a CRM.
You’re building a memory surface.

⸻

Bottom line

You’re already circling the right solution. I’d summarize it as:
	•	Global rules live in Settings
	•	Actual naming happens on the contact
	•	The Hero card is the canonical editing surface
	•	The chooser remains clean and focused
