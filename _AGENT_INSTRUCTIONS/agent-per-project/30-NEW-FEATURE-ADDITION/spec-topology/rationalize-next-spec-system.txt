You are working on a new branch. Your task is to refactor the spec family topology logic (spec → child spec resolution) so that it scales as the application grows, without changing observable behavior.

Background / Principles (read carefully)

This app uses spec systems (e.g. sidebar cassette specs, onboarding specs, tooltips specs) to avoid top-down complexity.
The key ideas you must preserve:
	1.	Specs avoid global complexity
	•	The app and the user focus on one spec at a time
	•	There is no giant “settings tree” or “help index”
	•	Navigation emerges from local parent → child relationships
	2.	Topology is global, meaning is local
	•	The existence of parent → child relationships is a domain-level concern
	•	The interpretation of a spec (how to render it) is a feature-level concern
	•	Features must not reason about other features
	3.	Features must not coordinate other features
	•	The Contacts feature must never decide “Messages comes next”
	•	Cross-feature transitions belong in essentials/domain, not in features
	4.	Extension methods on spec families are intentional
	•	childSpec() / nextSpec() logic lives with the spec family
	•	This keeps decisions small, local, and DDD-friendly
	•	This is not accidental coupling — it is topology definition
	5.	The problem
	•	Spec families (e.g. CassetteSpec) will grow
	•	A single monolithic cassette_spec.dart with all child logic will become hard to maintain
	•	We need physical modularization without re-centralizing logic or introducing a “God router”

Non-negotiable constraints
	•	Do NOT move child-spec decision logic into feature folders
	•	Do NOT introduce any top-down registry, database, or config table
	•	Do NOT make features aware of other features
	•	Do NOT change public APIs used by the rest of the app
	•	Do NOT change runtime behavior

What “good” looks like
	•	There is still one public entry point:
	•	e.g. CassetteSpec.childSpec()
	•	The implementation of child resolution is split across multiple files
	•	Each file is small, coherent, and locally understandable
	•	Cross-feature transitions are explicit and discoverable
	•	Adding new specs does not require editing a massive file
