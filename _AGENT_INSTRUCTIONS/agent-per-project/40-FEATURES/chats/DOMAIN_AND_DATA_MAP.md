# Domain & Data Map — Chats

## Core Entities
- `Chat` aggregate in `working.db` (`working.chats`).
- Participant join table `working.chat_to_participant`.

## Supporting Tables & Views
| Database | Table/View | Purpose | Notes |
| --- | --- | --- | --- |
| `db-import` | `chats` | Source chat metadata from `chat.db`. | Immutable ledger rows.
| `db-import` | `chat_message_join` | Connects chats to messages in ledger. | Input for migration sequencing.
| `db-working` | `chats` | Projection consumed by UI and search. | Maintains derived columns.
| `db-working` | `chat_participants_view` | Convenience view for resolving roster. | Generated by migrators.
| `db-overlay` | `chat_preferences` | User pin/archive/mute overrides. | Merges at presentation layer.

## External Inputs
- Import orchestrator populating ledger tables.
- Overlay services for pinned/archived state.

## Downstream Consumers
- Chat list UI, chat detail panels.
- Search feature (chat filters, jump-to-chat).
- Messages feature (chat header context).

## Data Contracts
- Chat GUID must remain stable across imports.
- Participant roster references canonical handle IDs.
- Derived counters (unread, attachment flags) must rebuild safely when migrations rerun.
